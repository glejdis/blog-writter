# Structural patterns (shared reference)

A reference the Stylist always reads before the example posts. These are the
structural shapes a technical post can take. When analyzing the corpus, name the
pattern(s) the examples actually use and tell the Writer when to reach for each.

## Top-down (general → specific)

Start with the big picture or the conclusion, then drill into details. Good for
readers who want the "what and why" before the "how". The default for reference
and architecture posts.

## Bottom-up (specific → general)

Build from concrete details, examples, or a single problem up to the broader
principle. Works well when the conclusion is surprising and you want the reader to
"discover" it with you.

## Inverted pyramid (borrowed from journalism)

Most important info first, then supporting detail, then nice-to-have context.
Readers can stop at any point and still have the key takeaway. Great for reference
posts and announcements.

## Problem → Solution

State a pain point, build tension, then resolve it. The classic "here's the bug
that cost us a weekend" post. Very engaging because it has narrative stakes.

## Chronological / narrative (the "journey")

Walk through what happened in order: "First I tried X, it broke, then Y." Honest
and relatable, but can ramble if you don't trim the dead ends.

## Tutorial / step-by-step

Sequential, imperative, reproducible. Each step builds on the last toward a
working result. PoC-driven posts lean this way.

## Compare and contrast

Option A vs Option B vs Option C, across fixed criteria. Good for "which
database / framework / pattern should I pick" posts. Usually ends with a
recommendation or a decision table.
